Bank System

Overview

The Bank System project is a Java application designed to simulate a banking system, allowing users to perform various banking operations such as depositing funds, withdrawing funds, transferring money between accounts, updating account information, and canceling accounts.

Design Principles

  • Best Programming Practices:
    • Utilization of try-catch statements to handle invalid user inputs.
    • Proper exception handling when dealing with file operations.
    • Consistent use of Javadoc comments to describe methods and classes.
  • Object-Oriented Programming (OOP) Concepts:
    • Encapsulation: Data hiding and protection through private member variables and public accessor methods.
    • Inheritance: Hierarchical organization of classes with common attributes and behaviors.
    • Polymorphism: Methods exhibiting different behavior based on the object they are called upon.
    • Interfaces: Contracts defining a set of methods that implementing classes must override.

Bank System Hierarchy Files

The Bank System is composed of three primary files. Each file contains and implements a set of banking operations as shown below:

App.java File

The App class responsible for managing user interactions and file operations. This file contains and implements the following methods:

  • loadUserData: This method is only called upon running the app. It loads user data from a CSV file (permanent storage) into the temporary storage of the application . If no CSV file exists, the application starts with an empty user list.
  • saveUserData: This method is only called upon closing the app. It saves user data to the permanent storage (CSV file).
  • generateRoutingNumber: This method is to generate a unique routing number for a new user. Bank users must have unique routing numbers associated their accounts.
    • A routing number is composed of 10 random digits, and it is randomly generated by the system.
    • A routing number may start with zeros (e.g., 0007742318). That means routing numbers should be treated as strings to avoid discarding zeros at the beginning if they were to be treated as numbers.
  • signUp: This method is to allow new users to create an account.
    • Users can register by providing their full name, username, password, email, and initial deposit amount. Each account is assigned a unique routing number.
    • An arrayList (temporary storage) should be used to store the user data during the runtime. The elements of the arrayList are the records in the CSV file (permanent storage).
    • Users must have unique emails. An email is composed of user's username followed by @bank.com (e.g., jsmith@bank.com).
  • logIn: This method is to allow existing users to log in and perform banking operations.
  • main: This method is the entry point of the application.

User.java File

The User class representing a bank user, implementing the BankActions interface. The User class is also used as a base class of another class (subclass) called RegularUser.

The User class file contains and implements the following methods:

  • constructor: Initializes user attributes including full name, username, password, email, routing number, checking balance, and saving balance.
  • getters and setters: Accessors and mutators for all user attributes.
  • home: Displays user's account information page.
  • deposit: Allows user to deposit funds into their account.
    • Each user in the system possesses both checking and savings accounts.
    • When depositing into the checking account including the initial deposit when signing up:
      • 2% of the deposit amount is added to the savings account balance.
      • The remaining amount (98%) is added to the checking account balance.
  • withdraw: Allows user to withdraw funds from their account.
    • When withdrawing from the savings account:
      • 5% of the withdrawal amount is designated as a withdrawal fee.
      • Ensures that the withdrawal amount plus the withdrawal fee does not exceed the savings account balance.
      • The withdrawal amount plus the withdrawal fee is deducted from the savings account balance.
  • update: Allows user to update their account information.
    • Users can only update their full names and passwords.
  • transfer: Allows user to transfer funds to another account.
    • Users can transfer money only to accounts within the same bank (internal accounts). For example, if a user attempts to transfer money to an account with the email 'example@gmail.com', the system will generate an error message. The message will state that the recipient's email must be associated with the bank's domain, such as '@bank', rather than an external domain like '@gmail'.
    • By default, all money transfers are done on checking accounts.
  • cancel: Allows user to cancel (delete) their account.

The User class file also contains and implements the RegularUser class. The RegularUser class is a subclass of the User class (inherited from the User class) representing regular bank users. The purpose of using the RegularUser class is to give users different account priviliages. In this case, all users have regular account privileges. The RegularUser class contains and implements only one method:

  • constructor: Calls the constructor of the superclass (User) to initialize user attributes. It represents bank users with regular account privileges, inheriting behaviors from the User class.

BankActions.java File

An interface file defining the core banking actions that must be implemented by the User class. The BankActions interface must have the following abstract methods declared:

  • home: Displays user's account information page.
  • deposit: Allows user to deposit funds into their account.
  • withdraw: Allows user to withdraw funds from their account.
  • update: Allows user to update their account information.
  • transfer: Allows user to transfer funds to another internal account.
  • cancel: Allows user to cancel (delete) their account.

Important Notes

Your implementation must adhere to the following programming requirements:

  • On the 'Bank System Page', a user can select an option by typing the choice in either uppercase or lowercase.
  • The temporary storage (ArrayList) must hold items of type User objects. Therefore, the ArrayList should be declared as ArrayList<User>, where User is a class.
  • Your implementation of the transfer and withdraw methods must handle 'insufficient funds' scenarios as specified in the sample output.
  • If a user chooses to cancel their account, they should be redirected to the 'Bank System Page' after successful deletion, as specified in the sample output.
  • When copying (writing) data from the temporary storage (ArrayList) to the permanent storage (CSV file), the original data in the CSV file must be overwritten. To achieve this, use StandardOpenOption.TRUNCATE_EXISTING when importing the StandardOpenOption class from the nio package as java.nio.file.StandardOpenOption.

Submission

  • Students may choose to demonstrate their final project to the professor before submission to ensure its functionality. A zoom-based or face-to-face meetings can be arranaged for this purpose.
  • Late submission will not be accepted for the final project.
  • Students are required to turn in the final project files, which must be ".java" files. They may also choose to compress their entire project package and submit it as a compressed file (e.g., .zip, .rar).
